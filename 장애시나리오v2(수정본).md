## 1. 역할 분담 (2인 테스트)

| 역할 | 이름 | 담당 |
| --- | --- | --- |
| **Attacker (A)** | VM 호스트 보유자 | 서비스 stop, 네트워크 차단, kill 등 장애 유발 |
| **Observer (B)** | 원격 접속자 | 상태 쿼리, 트래픽 발생, 결과 기록 |

> 💡 각 단계마다 **A, B 라벨** 로 누가 실행할지 표시합니다.

## 2. 사전 준비

### 2.1 [공통] 모든 VM에 기본 도구 설치

💻 **[어디서?]** 모든 VM (각각 SSH 접속해서)
🖥️ **[쉘은?]** bash
📦 **[설치]**

```bash
sudo apt update
sudo apt install -y curl jq tmux watch iptables-persistent
```

### 2.2 [PXC 노드] 모니터링 계정 생성

💻 **[어디서?]** `pxc-01` 한 곳에서만 (자동 복제됨)
🖥️ **[쉘은?]** 먼저 bash → mysql 쉘로 진입

▶ **[실행]**
```bash
sudo mysql -u root -p
```

프롬프트가 `mysql>` 로 바뀌면:

```sql
CREATE USER 'monitor'@'%' IDENTIFIED BY 'MonPass123!';
GRANT PROCESS, REPLICATION CLIENT ON *.* TO 'monitor'@'%';
FLUSH PRIVILEGES;
EXIT;
```

다시 bash 에서 `monitor` 계정용 설정파일 생성 (**pxc-01, pxc-02, pxc-03 모두에서**):

```bash
cat > ~/.my-monitor.cnf << 'EOF'
[client]
user=monitor
password=MonPass123!
host=127.0.0.1
EOF

chmod 600 ~/.my-monitor.cnf
```

✅ **[확인]**
```bash
mysql --defaults-file=~/.my-monitor.cnf -e "SELECT 1;"
```
→ `1` 이 출력되면 성공.

### 2.3 [RabbitMQ 노드] Python + pika 설치

💻 **[어디서?]** `rabbit-1`, `rabbit-2`, `rabbit-3` 전부 (또는 테스트 돌릴 곳만)

📦 **[설치]**

### 2.4 [Redis 노드] redis-cli 확인

### 2.5 [HAProxy 노드] socat + stats 페이지 활성화

💻 **[어디서?]** `haproxy-1`, `haproxy-2`
🖥️ **[쉘은?]** bash

📦 **[설치]**
```bash
sudo apt install -y socat
```

**stats 웹 페이지 켜기** — `/etc/haproxy/haproxy.cfg` 편집:

```bash
sudo nano /etc/haproxy/haproxy.cfg
```

파일 **맨 아래에** 다음 블록 추가:

```
listen stats
    bind *:8404
    mode http
    stats enable
    stats uri /stats
    stats refresh 3s
    stats admin if TRUE
    stats auth admin:StatsPass123!
```

저장(`Ctrl+O`, Enter, `Ctrl+X`) 후:
```bash
sudo systemctl reload haproxy
```

✅ **[확인]** Mac 브라우저에서 열기:
```
http://<haproxy-1 IP>:8404/stats
```
(로그인 계정: admin / StatsPass123!)

## 3. PXC (MySQL) 장애 테스트

### 3.1 정상 상태 확인

💻 **[어디서?]** 아무 PXC 노드 (예: `pxc-01`)
🖥️ **[쉘은?]** bash

▶ **[실행]**
```bash
mysql --defaults-file=~/.my-monitor.cnf -e "
SHOW STATUS WHERE Variable_name IN (
  'wsrep_cluster_size',
  'wsrep_cluster_status',
  'wsrep_local_state_comment',
  'wsrep_ready',
  'wsrep_connected'
);"
```

✅ **[기대결과]**
```
+---------------------------+---------+
| wsrep_cluster_size        | 3       |
| wsrep_cluster_status      | Primary |
| wsrep_local_state_comment | Synced  |
| wsrep_ready               | ON      |
| wsrep_connected           | ON      |
+---------------------------+---------+
```

### 3.2 실시간 모니터링 시작

💻 **[어디서?]** `pxc-01` (Observer B가 접속)
🖥️ **[쉘은?]** bash

▶ **[실행]**
```bash
watch -n 1 'mysql --defaults-file=~/.my-monitor.cnf -e "
SHOW STATUS WHERE Variable_name IN (
  \"wsrep_cluster_size\",\"wsrep_cluster_status\",
  \"wsrep_local_state_comment\",\"wsrep_incoming_addresses\"
);"'
```

🛑 **[종료방법]** `Ctrl+C`

### 3.3 테스트 데이터 준비

💻 **[어디서?]** `pxc-01`
🖥️ **[쉘은?]** bash → mysql

▶ **[실행]**
```bash
sudo mysql -u root -p
```

```sql
CREATE DATABASE IF NOT EXISTS hatest;
USE hatest;
CREATE TABLE IF NOT EXISTS t (
  id INT AUTO_INCREMENT PRIMARY KEY,
  v VARCHAR(50),
  node VARCHAR(20),
  ts DATETIME DEFAULT CURRENT_TIMESTAMP
);
INSERT INTO t (v, node) VALUES ('before-failure', @@hostname);
SELECT * FROM t;
EXIT;
```

### 3.4 시나리오 PXC-1 — 노드 1대 정상 종료

**목적**: graceful shutdown 시 quorum 유지 확인

👤 **A**
💻 **[어디서?]** `pxc-02`
```bash
sudo systemctl stop mysql
```

👤 **B**
💻 **[어디서?]** `pxc-01` (watch 돌아가는 창)

✅ **[기대결과]**
- `wsrep_cluster_size`: 3 → 2
- `wsrep_cluster_status`: **Primary** 유지
- 쓰기 가능 (테스트: `INSERT INTO hatest.t (v,node) VALUES ('during',@@hostname);`)

🧹 **[복구]**
👤 **A** `pxc-02` 에서:
```bash
sudo systemctl start mysql
```
→ 자동으로 IST 동기화. 30초~1분 내 Synced 복귀.

### 3.5 시나리오 PXC-2 — 노드 2대 "정상" 종료 (Primary 유지)

👤 **A**
💻 **[어디서?]** `pxc-02`, `pxc-03` 각각
```bash
sudo systemctl stop mysql
```

✅ **[기대결과]** `pxc-01` 에서:
- `wsrep_cluster_size`: 1
- `wsrep_cluster_status`: **Primary** (⚠️ 의도된 동작!)

> 💡 **중요**: `systemctl stop` 은 graceful leave 라서 quorum 이 재조정됩니다. Non-Primary 재현은 3.6 참고.

🧹 **[복구]** `pxc-02`, `pxc-03` 순서대로:
```bash
sudo systemctl start mysql
```

### 3.6 시나리오 PXC-3 — 노드 2대 **강제** 장애 (Non-Primary 재현)

👤 **A**
💻 **[어디서?]** `pxc-02`, `pxc-03` 각각
▶ **[실행]** 네트워크 차단으로 "갑자기 연락 끊긴" 상태 시뮬레이션:
```bash
# Galera 통신 포트 막기
sudo iptables -A INPUT  -p tcp --dport 4567 -j DROP
sudo iptables -A OUTPUT -p tcp --dport 4567 -j DROP
```

👤 **B**
💻 **[어디서?]** `pxc-01`

✅ **[기대결과]** (수초~수십초 후)
- `wsrep_cluster_size`: 1
- `wsrep_cluster_status`: **Non-Primary**
- 쓰기 시도 시 에러:
  ```
  ERROR 1047 (08S01): WSREP has not yet prepared node for application use
  ```

🧹 **[복구]**
👤 **A** `pxc-02`, `pxc-03` 에서:
```bash
sudo iptables -F   # 전체 규칙 초기화
```

`pxc-01` 이 Primary 로 자동 복귀되는지 확인. 안 되면:
```bash
# pxc-01 에서
mysql --defaults-file=~/.my-monitor.cnf -e "SET GLOBAL wsrep_provider_options='pc.bootstrap=YES';"
```

### 3.7 시나리오 PXC-4 — 전체 클러스터 다운 후 복구

👤 **A**
💻 **[어디서?]** 3대 모두 순차적으로
```bash
sudo systemctl stop mysql
```

**복구 절차:**

① **각 노드에서 seqno 확인**
```bash
sudo cat /var/lib/mysql/grastate.dat
```

`seqno` 값이 `-1` 이면 비정상 종료:
```bash
sudo mysqld --wsrep-recover 2>&1 | grep -i "Recovered position"
```

② **seqno 가 가장 큰 노드** 에서 bootstrap 플래그 세팅:
```bash
sudo vi /var/lib/mysql/grastate.dat
# safe_to_bootstrap: 1 로 변경 후 저장
```

③ 같은 노드에서 bootstrap 으로 시작:
```bash
sudo systemctl start mysql@bootstrap.service
```

④ 나머지 노드는 일반 start:
```bash
sudo systemctl start mysql
```

✅ **[확인]** 전체 복구 후:
```bash
mysql --defaults-file=~/.my-monitor.cnf -e "SHOW STATUS LIKE 'wsrep_cluster_size';"
# → 3
```

### 4.3 Publisher 스크립트 작성

💻 **[어디서?]** `rabbit-1` (또는 어느 rabbit 노드든)
🖥️ **[쉘은?]** bash
📦 **[설치]** 2.3 절에서 이미 준비됨 (venv + pika)

▶ **[실행]** — 파일 생성:
```bash
cd ~/rmq-test
source venv/bin/activate    # 가상환경 켜기 (프롬프트에 (venv) 보여야 함)

cat > publisher.py << 'EOF'
import pika, time, sys

# 연결할 RabbitMQ 호스트 (인자로 받거나 기본값)
host = sys.argv[1] if len(sys.argv) > 1 else 'localhost'
total = int(sys.argv[2]) if len(sys.argv) > 2 else 1000

conn = pika.BlockingConnection(pika.ConnectionParameters(host))
ch = conn.channel()
ch.queue_declare(queue='hatest', durable=True)

print(f"Publishing {total} messages to {host}...")
for i in range(total):
    ch.basic_publish(
        exchange='',
        routing_key='hatest',
        body=f'msg-{i}'.encode(),
        properties=pika.BasicProperties(delivery_mode=2),  # persistent
    )
    print(f"  sent msg-{i}")
    time.sleep(0.2)

conn.close()
print("Done.")
EOF
```

▶ **[실행]**
```bash
# 같은 노드(localhost)로 보내기, 100개 메시지
python3 publisher.py localhost 100
```

🛑 **[종료방법]** 중간에 멈추려면 `Ctrl+C`

### 4.4 Consumer 스크립트 작성

💻 **[어디서?]** `rabbit-2` (다른 노드에서 받아봄)
🖥️ **[쉘은?]** bash

```bash
cd ~/rmq-test
source venv/bin/activate

cat > consumer.py << 'EOF'
import pika, sys

host = sys.argv[1] if len(sys.argv) > 1 else 'localhost'

conn = pika.BlockingConnection(pika.ConnectionParameters(host))
ch = conn.channel()
ch.queue_declare(queue='hatest', durable=True)

count = 0
def on_msg(ch, method, props, body):
    global count
    count += 1
    print(f"[{count}] {body.decode()}")
    ch.basic_ack(method.delivery_tag)

print(f"Consuming from {host} (Ctrl+C to stop)...")
ch.basic_consume(queue='hatest', on_message_callback=on_msg)
try:
    ch.start_consuming()
except KeyboardInterrupt:
    print(f"\nTotal consumed: {count}")
    ch.stop_consuming()
    conn.close()
EOF
```

▶ **[실행]**
```bash
python3 consumer.py localhost
```

🛑 **[종료방법]** `Ctrl+C` — 종료 시 총 수신 메시지 수가 출력됨.

### 4.5 시나리오 RMQ-1 — Publisher 돌리는 중 노드 다운

**준비**: 3개의 터미널 필요
- 터미널 1 (B): rabbit-2 에서 consumer 실행
- 터미널 2 (B): rabbit-1 에서 publisher 실행
- 터미널 3 (A): rabbit-1 다운 시키는 용

**순서:**

① **B (터미널 1)** `rabbit-2` 에서:
```bash
cd ~/rmq-test && source venv/bin/activate
python3 consumer.py localhost
```

② **B (터미널 2)** `rabbit-1` 에서:
```bash
cd ~/rmq-test && source venv/bin/activate
python3 publisher.py localhost 200
```

③ **A (터미널 3)** `rabbit-1` 에서 (publisher가 중간쯤 진행됐을 때):
```bash
sudo rabbitmqctl stop_app
```

✅ **[기대결과]**
- publisher 가 중간에 에러 (연결 끊김)
- consumer 는 계속 수신 (rabbit-2 살아있음)
- `sudo rabbitmqctl cluster_status` → Running Nodes: 2개

🧹 **[복구]**
```bash
# rabbit-1 에서
sudo rabbitmqctl start_app
```

## 5. Redis 장애 테스트

### 5.1 정상 상태 확인 먼저 — 어떤 모드인지 파악

💻 **[어디서?]** `redis-01`
🖥️ **[쉘은?]** bash

**Sentinel 모드인지 확인:**
```bash
ps aux | grep -E 'redis-sentinel|sentinel.conf' | grep -v grep
```

**Cluster 모드인지 확인:**
```bash
redis-cli -p 6379 cluster info 2>&1 | head -3
```

→ `cluster_enabled:1` 이면 Cluster 모드, 아니면 Sentinel 또는 단순 Replication.

### 5.2 Replication/Sentinel 모드 — 현재 마스터 찾기

💻 **[어디서?]** 아무 redis 노드
```bash
# 각 노드의 역할 확인
redis-cli -p 6379 info replication | grep role
```

Sentinel 이 있다면:
```bash
redis-cli -p 26379 sentinel master mymaster | head -20
redis-cli -p 26379 sentinel get-master-addr-by-name mymaster
```
→ 현재 마스터 IP/포트가 출력됨.

### 5.3 테스트 데이터 넣기

💻 **[어디서?]** Observer (B)가 접속한 곳
```bash
# 현재 마스터가 redis-01 이라 가정
redis-cli -h redis-01 -p 6379 SET ha:test1 "before-failover"
redis-cli -h redis-01 -p 6379 GET ha:test1
```

### 5.4 모니터링 시작

💻 **[어디서?]** `redis-02` (곧 마스터로 승격될 가능성 있는 노드)
```bash
watch -n 1 'redis-cli -p 6379 info replication | grep -E "role|master_host|master_port|connected_slaves"'
```

🛑 **[종료방법]** `Ctrl+C`

### 5.5 시나리오 REDIS-1 — 마스터 강제 종료

👤 **A**
💻 **[어디서?]** 현재 마스터 (예: `redis-01`)
```bash
sudo systemctl stop redis
# 또는 더 급하게:
# sudo pkill -9 redis-server
```

👤 **B**
💻 **[어디서?]** `redis-02` 의 watch 창

✅ **[기대결과]** (Sentinel 기준 보통 30초 내)
- 어느 한 replica 가 `role:master` 로 승격
- 다른 replica 들이 새 마스터를 `master_host` 로 잡음

승격 후 데이터 확인:
```bash
# 새 마스터(IP 확인 후)에서
redis-cli -h <새마스터IP> -p 6379 GET ha:test1
# "before-failover" 가 나와야 함

# 새 마스터에 쓰기도 가능한지
redis-cli -h <새마스터IP> -p 6379 SET ha:test2 "after-failover"
```

🧹 **[복구]**
👤 **A** 원래 마스터였던 노드에서:
```bash
sudo systemctl start redis
```
→ 자동으로 replica 로 재조인.

## 9. 트러블슈팅

| 증상 | 원인 | 해결 |
| --- | --- | --- |
| PXC: `Access denied for user 'root'@'localhost'` | root 비번 필요 | `sudo mysql -u root -p` 로 접속 |
| PXC: `cluster_status: Non-Primary` 계속 | quorum 상실 | 다른 노드 복구 or `pc.bootstrap=YES` |
| PXC: `It may not be safe to bootstrap` | `safe_to_bootstrap:0` | seqno 비교 후 `1` 로 수동 변경 |
| pika: `ModuleNotFoundError: No module named 'pika'` | venv 안 켬 | `source venv/bin/activate` |
| pika: `ConnectionRefusedError` | RabbitMQ 다운 or 포트 | `sudo systemctl status rabbitmq-server` |
| HAProxy: `socat: command not found` | socat 미설치 | `sudo apt install -y socat` |
| HAProxy stats: 브라우저 접근 불가 | 방화벽 / stats 블록 미작성 | cfg 확인, 8404 포트 확인 |
| redis-cli: `NOAUTH Authentication required` | 비밀번호 필요 | `redis-cli -a <비밀번호>` |
| iptables 복구 후 통신 안 됨 | 룰이 저장돼 있음 | `sudo iptables -F && sudo netfilter-persistent save` |

## 📎 부록 A — 자주 묻는 질문

**Q1. SQL 명령을 bash 에 쳤더니 `command not found` 뜨는데요?**
A: SQL 은 `mysql>` 프롬프트 안에서 실행해야 해요. bash 에서 실행하려면 `mysql -e "SQL 명령"` 형식으로.

**Q2. `pika` import 안 되는데요?**
A: `source ~/rmq-test/venv/bin/activate` 해서 가상환경 켜야 합니다. 프롬프트 앞에 `(venv)` 가 보여야 함.

**Q3. `systemctl stop mysql` 해도 Primary 유지되는데 버그 아닌가요?**
A: graceful leave 는 클러스터 멤버십 재조정이라 Primary 유지가 정상. 실제 장애(네트워크 단절 등) 시나리오는 iptables 방식 사용.

**Q4. HAProxy 의 "backend 서비스" 가 뭔가요?**
A: HAProxy가 트래픽을 뒤로 보내는 실제 애플리케이션 서버예요. backend-1, backend-2 VM 에서 돌고 있는 웹서버/API서버. 실제 서비스 이름은 `sudo systemctl list-units --type=service | grep -v '\.service.*loaded active'` 로 확인.

**Q5. 두 명이 같은 터미널 화면을 보고 싶어요.**
A: 한 명이 `tmux new -s work`, 다른 한 명이 같은 VM 에 SSH 접속 후 `tmux attach -t work`.
