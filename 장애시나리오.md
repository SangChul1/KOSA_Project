# HA 클러스터 장애 시나리오 & 모니터링 가이드

> **대상 인프라**
> - HAProxy: `haproxy-1`, `haproxy-2`
> - Backend: `backend-1`, `backend-2`
> - RabbitMQ: `rabbit-1`, `rabbit-2`, `rabbit-3`
> - Percona XtraDB Cluster: `pxc-01`, `pxc-02`, `pxc-03`
> - Redis: `redis-01`, `redis-02`, `redis-03`
>
> **목표**: 각 컴포넌트에 장애를 유발하고, 서비스 가용성 / 데이터 일관성 / 자동 페일오버가 정상 동작하는지 2인 1조로 검증한다.

---

## 0. 사전 준비

### 0.1 공통 — 역할 분담

| 역할 | 담당자 | 하는 일 |
| --- | --- | --- |
| **Attacker (A)** | VM 호스트 (Mac) 보유자 | 노드/서비스 kill, 네트워크 차단 등 장애 유발 |
| **Observer (B)** | 원격 SSH 접속자 | 상태 쿼리, 트래픽 발생, 로그 모니터링 |

### 0.2 필수 도구 설치

각 VM 공통:

```bash
sudo apt update
sudo apt install -y socat tmux curl jq watch
```

### 0.3 MySQL 모니터링 전용 계정 생성 (PXC 한 노드에서만)

```sql
CREATE USER 'monitor'@'%' IDENTIFIED BY 'MonPass123!';
GRANT PROCESS, REPLICATION CLIENT ON *.* TO 'monitor'@'%';
FLUSH PRIVILEGES;
```

`~/.my-monitor.cnf` 생성:
```ini
[client]
user=monitor
password=MonPass123!
```
```bash
chmod 600 ~/.my-monitor.cnf
```

### 0.4 HAProxy stats 페이지 활성화

`/etc/haproxy/haproxy.cfg` 에 다음 블록 추가:

```
listen stats
    bind *:8404
    mode http
    stats enable
    stats uri /stats
    stats refresh 3s
    stats admin if TRUE
    stats auth admin:StatsPass123!
```

```bash
sudo systemctl reload haproxy
```

브라우저에서 `http://<haproxy-1 IP>:8404/stats` 접속 가능.

### 0.5 공유 작업 환경 (tmux)

여러 VM을 한 화면에서 보려면:

```bash
# A가 먼저
tmux new -s ha-test

# B는
tmux attach -t ha-test
```

창 분할: `Ctrl+b` → `%` (세로), `Ctrl+b` → `"` (가로), `Ctrl+b` → 방향키 (이동)

## 1. PXC (MySQL 클러스터) 장애 테스트

### 1.1 정상 상태 확인

```bash
mysql --defaults-file=~/.my-monitor.cnf -e "
SHOW STATUS WHERE Variable_name IN (
  'wsrep_cluster_size',
  'wsrep_cluster_status',
  'wsrep_local_state_comment',
  'wsrep_ready',
  'wsrep_connected',
  'wsrep_flow_control_paused',
  'wsrep_cluster_state_uuid',
  'wsrep_incoming_addresses'
);"
```

**기대값**
| 항목 | 정상값 |
| --- | --- |
| wsrep_cluster_size | 3 |
| wsrep_cluster_status | Primary |
| wsrep_local_state_comment | Synced |
| wsrep_ready | ON |
| wsrep_connected | ON |
| wsrep_flow_control_paused | 0.0 |

### 1.2 실시간 모니터링 (B가 실행)

```bash
watch -n 1 'mysql --defaults-file=~/.my-monitor.cnf -e "
SHOW STATUS WHERE Variable_name IN (
  \"wsrep_cluster_size\",
  \"wsrep_cluster_status\",
  \"wsrep_local_state_comment\",
  \"wsrep_incoming_addresses\"
);"'
```

### 1.3 시나리오 PXC-1 — 노드 1대 다운

**A 실행:**
```bash
# pxc-02 접속 후
sudo systemctl stop mysql
```

**B 확인:**
- `wsrep_cluster_size`: 3 → 2
- `wsrep_cluster_status`: `Primary` 유지 (quorum 유지)
- 남은 2개 노드에 쓰기 여전히 가능

**복구 (A 실행):**
```bash
sudo systemctl start mysql
```
→ IST(Incremental State Transfer)로 자동 재동기화. `cluster_size` 가 3으로 복귀.

### 1.4 시나리오 PXC-2 — 노드 2대 다운 (quorum 상실)

**A 실행:**
```bash
# pxc-02, pxc-03 두 대 stop
sudo systemctl stop mysql   # 각각에서
```

**B 확인:**
- 남은 `pxc-01` 의 `wsrep_cluster_status` → `Non-Primary`
- 쿼리 실행 시 에러: `WSREP has not yet prepared node for application use`
- **쓰기 불가 상태** (의도된 동작, split-brain 방지)

**복구:**
```bash
# pxc-02, pxc-03 순서대로 start (bootstrap 아님)
sudo systemctl start mysql
```
→ quorum 복구되면 자동으로 Primary 상태 회복.

### 1.5 시나리오 PXC-3 — 전체 클러스터 다운

**A 실행:**
```bash
# 3대 모두 stop
sudo systemctl stop mysql
```

**복구 절차 (가장 중요한 시나리오):**

1. 각 노드에서 마지막 seqno 확인:
   ```bash
   cat /var/lib/mysql/grastate.dat
   # seqno 값이 -1 이면:
   sudo mysqld_safe --wsrep-recover
   sudo grep -i "Recovered position" /var/log/mysql/error.log | tail -5
   ```

2. **seqno 가 가장 큰 노드** 선택 후:
   ```bash
   sudo vi /var/lib/mysql/grastate.dat
   # safe_to_bootstrap: 1 로 변경
   ```

3. 해당 노드에서 bootstrap:
   ```bash
   sudo systemctl start mysql@bootstrap.service
   ```

4. 나머지 노드는 일반 start:
   ```bash
   sudo systemctl start mysql
   ```

### 1.6 시나리오 PXC-4 — 데이터 일관성 검증

**B 실행 (테스트 데이터 준비):**
```sql
CREATE DATABASE IF NOT EXISTS hatest;
USE hatest;
CREATE TABLE IF NOT EXISTS t (id INT AUTO_INCREMENT PRIMARY KEY, v VARCHAR(50), node VARCHAR(20), ts DATETIME DEFAULT CURRENT_TIMESTAMP);
INSERT INTO t (v, node) VALUES ('before-failure', @@hostname);
```

**A 실행:**
```bash
# pxc-03 다운시킴
sudo systemctl stop mysql
```

**B 실행 (장애 중 쓰기):**
```sql
INSERT INTO hatest.t (v, node) VALUES ('during-failure', @@hostname);
```

**복구 후 검증:**
```sql
-- pxc-03 복구 후 이 노드에서 확인
SELECT * FROM hatest.t ORDER BY id;
-- 장애 중 입력된 데이터도 정상 복제되어 있어야 함
```

## 2. RabbitMQ 클러스터 장애 테스트

### 2.1 정상 상태 확인

```bash
sudo rabbitmqctl cluster_status
sudo rabbitmqctl list_queues name messages consumers
```

**기대 출력**
- `Running Nodes`: 3개 노드 모두 표시
- `Partitions`: 비어있음

### 2.2 실시간 모니터링

```bash
watch -n 2 'sudo rabbitmqctl cluster_status | head -40'
```

Management UI (15672 포트가 열려있다면):
```
http://<rabbit-1 IP>:15672
```

### 2.3 시나리오 RMQ-1 — 노드 1대 다운

**준비 (큐가 HA/mirrored 또는 quorum 큐인지 확인):**
```bash
sudo rabbitmqctl list_policies
sudo rabbitmqctl list_queues name type
```

**A 실행:**
```bash
# rabbit-2 에서
sudo rabbitmqctl stop_app
```

**B 확인:**
```bash
sudo rabbitmqctl cluster_status
# Running Nodes: 2개로 줄어듦
sudo rabbitmqctl list_queues name messages
# 메시지 유지됨
```

**복구:**
```bash
sudo rabbitmqctl start_app
```

### 2.4 시나리오 RMQ-2 — 메시지 손실 없음 검증

**B 실행 (publisher):**
```python
# publisher.py
import pika, time
conn = pika.BlockingConnection(pika.ConnectionParameters('rabbit-1'))
ch = conn.channel()
ch.queue_declare(queue='hatest', durable=True)
for i in range(1000):
    ch.basic_publish(
        exchange='',
        routing_key='hatest',
        body=f'msg-{i}'.encode(),
        properties=pika.BasicProperties(delivery_mode=2))
    time.sleep(0.1)
    print(f"sent {i}")
```

**A 실행 (도중에 rabbit-1 다운):**
```bash
sudo rabbitmqctl stop_app
```

**B 실행 (consumer, rabbit-2 에서):**
```python
# consumer.py
import pika
conn = pika.BlockingConnection(pika.ConnectionParameters('rabbit-2'))
ch = conn.channel()
ch.queue_declare(queue='hatest', durable=True)
def on_msg(ch, method, props, body):
    print(body.decode())
    ch.basic_ack(method.delivery_tag)
ch.basic_consume(queue='hatest', on_message_callback=on_msg)
ch.start_consuming()
```

**검증**: 전송한 메시지 수와 수신한 메시지 수가 일치해야 함 (mirrored/quorum queue 정책이 설정된 경우).

### 2.5 시나리오 RMQ-3 — 네트워크 파티션

**A 실행:**
```bash
# rabbit-3 에서 다른 두 노드 차단
sudo iptables -A INPUT -s <rabbit-1 IP> -j DROP
sudo iptables -A INPUT -s <rabbit-2 IP> -j DROP
```

**B 확인:**
```bash
sudo rabbitmqctl cluster_status
# Partitions 섹션에 내용 표시됨 (split-brain 감지)
```

**복구:**
```bash
sudo iptables -F
sudo rabbitmqctl start_app   # partition handling 정책에 따라
```

## 3. Redis 클러스터 장애 테스트

### 3.1 정상 상태 확인

```bash
# Sentinel 모드인 경우
redis-cli -p 26379 sentinel master mymaster
redis-cli -p 26379 sentinel replicas mymaster

# Cluster 모드인 경우
redis-cli -p 6379 cluster nodes
redis-cli -p 6379 cluster info
```

### 3.2 실시간 모니터링

```bash
watch -n 1 'redis-cli -p 6379 info replication'
```

### 3.3 시나리오 REDIS-1 — 마스터 다운 (자동 페일오버)

**A 실행:**
```bash
# 현재 마스터(예: redis-01) stop
sudo systemctl stop redis
```

**B 확인:**
```bash
# Sentinel: 새 마스터로 승격되는 것 확인
redis-cli -p 26379 sentinel master mymaster
# 또는
redis-cli -p 26379 sentinel get-master-addr-by-name mymaster

# Cluster 모드: 슬롯 재할당 확인
redis-cli -p 6379 cluster nodes
```

**예상 소요 시간**: Sentinel 기본 down-after-milliseconds 30초 + 선거 시간.

**복구:**
```bash
sudo systemctl start redis
# 자동으로 replica 로 조인
```

### 3.4 시나리오 REDIS-2 — 데이터 유지 검증

**B 실행 (쓰기):**
```bash
redis-cli -h redis-01 SET k1 "before-failure"
redis-cli -h redis-01 GET k1
```

**A 실행:**
```bash
# redis-01 (현재 마스터) stop
sudo systemctl stop redis
```

**B 재확인 (새 마스터로 접속):**
```bash
redis-cli -h <새 마스터 IP> GET k1
# "before-failure" 가 나와야 함
redis-cli -h <새 마스터 IP> SET k2 "after-failover"
```

## 4. HAProxy 장애 테스트

### 4.1 정상 상태 확인

Stats 웹페이지에서 모든 backend `UP` 상태 확인:
```
http://<haproxy-1 IP>:8404/stats
```

CLI:
```bash
echo "show stat" | sudo socat stdio /var/run/haproxy.sock | column -s, -t | less -S
echo "show servers state" | sudo socat stdio /var/run/haproxy.sock
```

### 4.2 시나리오 HAP-1 — Backend 1대 다운

**B 실행 (연속 요청):**
```bash
while true; do
  echo "$(date +%T) $(curl -s http://<haproxy VIP 또는 haproxy-1>/health)"
  sleep 1
done
```

**A 실행:**
```bash
# backend-1 에서
sudo systemctl stop <backend 서비스>
```

**B 확인:**
- HAProxy stats 페이지: `backend-1` 빨간색(DOWN)
- curl 결과: 전부 `backend-2` 로 응답
- **요청 실패 없이 전환** 되어야 정상

**복구:**
```bash
sudo systemctl start <backend 서비스>
```
→ Health check 통과 후 자동으로 다시 UP.

### 4.3 시나리오 HAP-2 — HAProxy 자체 다운 (keepalived/VIP)

HAProxy가 VIP(예: keepalived) 구성이라면:

**A 실행:**
```bash
# haproxy-1 에서
sudo systemctl stop haproxy
```

**B 확인:**
- VIP가 `haproxy-2` 로 이동했는지 (`ip a` 로 확인)
- 서비스 요청이 끊김 없이 처리되는지

```bash
ping -c 5 <VIP>
curl -s http://<VIP>/health
```

## 5. 통합 End-to-End 장애 테스트

### 5.1 시나리오 E2E-1 — 전체 스택 부하 중 장애

**B 실행 (부하 발생):**
```bash
# 10분간 지속 요청
for i in $(seq 1 600); do
  curl -s -o /dev/null -w "%{http_code} %{time_total}s\n" http://<VIP>/api/test
  sleep 1
done
```

**A 실행 (1분마다 장애 유발):**
```bash
sleep 60; sudo systemctl stop mysql       # pxc-02
sleep 60; sudo systemctl start mysql
sleep 60; sudo systemctl stop rabbitmq-server   # rabbit-2
sleep 60; sudo systemctl start rabbitmq-server
sleep 60; sudo systemctl stop redis        # redis-01
sleep 60; sudo systemctl start redis
```

**검증 항목**
- HTTP 200 비율이 99% 이상 유지되는가
- p95 응답시간이 장애 중에도 허용 범위 내인가
- 장애 후 모든 데이터가 일관되게 복구되었는가

### 5.2 시나리오 E2E-2 — 동시 다중 장애 (극단적)

**A 실행:**
```bash
# 동시에 각 서비스 1대씩 다운
pxc-02, rabbit-2, redis-02, backend-1 동시 stop
```

**B 확인:**
- 서비스 여전히 정상 응답하는가 (quorum 유지되는 수준의 장애)
- 이후 순차적으로 복구되는지

## 7. 체크리스트

### 7.1 테스트 전

- [ ] HAProxy stats 페이지 접속 확인
- [ ] tmux 공유 세션 준비 (선택)
- [ ] 테스트 데이터 초기화

- [ ] 장애 유발 직전 정상 상태 스냅샷(쿼리 결과) 저장

- [ ] 복구 완료 시각 기록

- [ ] 모든 노드 정상 상태 복귀 확인
- [ ] `grastate.dat` 의 `safe_to_bootstrap` 값이 모두 `0` 인지 확인 (bootstrap 테스트했다면 원복)

- [ ] iptables 규칙 정리 (`sudo iptables -F` — 운영 환경이면 주의!)

## 8. 트러블슈팅 빠른 참조

| 증상 | 원인 | 해결 |
| --- | --- | --- |
| PXC: `ERROR 1047: WSREP has not yet prepared node` | quorum 상실 (Non-Primary) | 최소 2개 노드 복구 or bootstrap |
| PXC: `It may not be safe to bootstrap` | `safe_to_bootstrap: 0` | 최신 seqno 노드 확인 후 `1`로 수정 |
| PXC: SST 계속 실패 | 4444/4567/4568 포트 막힘 or SST 유저 권한 | 방화벽 / `wsrep_sst_auth` 확인 |
| RabbitMQ: `Partitions` 에 내용 표시 | split-brain | `pause_minority` 정책 or 수동 복구 |
| Redis: Sentinel이 페일오버 안 함 | quorum 미달 | Sentinel 수(quorum 값) 점검 |
| HAProxy: 모든 backend DOWN | health check 경로/포트 오류 | `haproxy.cfg` `option httpchk` 확인 |

## 부록 A. 자주 쓰는 명령 치트시트

```bash
# === PXC ===
mysql --defaults-file=~/.my-monitor.cnf -e "SHOW STATUS LIKE 'wsrep_%';"
sudo systemctl start mysql@bootstrap.service    # 첫 노드 부트스트랩
sudo mysqld_safe --wsrep-recover                # seqno 복구

# === RabbitMQ ===
sudo rabbitmqctl cluster_status
sudo rabbitmqctl stop_app / start_app
sudo rabbitmqctl list_queues name messages consumers

# === Redis ===
redis-cli -p 6379 info replication
redis-cli -p 26379 sentinel master mymaster
redis-cli -p 6379 cluster nodes

# === HAProxy ===
echo "show stat" | sudo socat stdio /var/run/haproxy.sock
echo "show servers state" | sudo socat stdio /var/run/haproxy.sock
sudo systemctl reload haproxy
```
